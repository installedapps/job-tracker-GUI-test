=== Final rating

`7.5 / 10`

This is a solid small GUI app. It is clearly beyond a throwaway beginner project because it already separates domain logic, UI rendering, persistence, and state. The biggest improvement area is not correctness, but making the UI theme/layout more centralised and making future features less likely to require editing many files at once.

== What the App Currently Does

The app is a local-first desktop job application tracker built with Rust, `iced`, and SQLite. It lets the user:

* Add job applications.
* Edit existing applications.
* Delete applications.
* Track company, role title, status, application date, and comments.
* Use a small calendar dropdown to choose a date.
* Switch between an Applications panel and a Stats panel.
* View status counts and progress bars.
* View a small heatmap of application activity.
* Export a Sankey-style SVG summary of the pipeline.
* Store data locally in a SQLite database.

== Architecture Summary

The structure is good because it avoids putting everything in `main.rs`.

[source,text]
----
src/
├── app.rs
├── app_actions.rs
├── app_defaults.rs
├── app_export.rs
├── error.rs
├── heatmap.rs
├── lib.rs
├── main.rs
├── message.rs
├── paths.rs
├── sankey.rs
├── stats.rs
├── model/
├── repository/
├── ui/
└── ui_state/
----

The app has three broad layers:

[cols="1,2,2", options="header"]
|===
|Layer |Main files |Purpose

|Application layer
|`app.rs`, `app_actions.rs`, `app_export.rs`, `message.rs`
|Holds app state, handles messages, performs save/edit/delete/export actions, and decides which view to show.

|Domain/data layer
|`model/`, `stats.rs`, `heatmap.rs`, `sankey.rs`
|Defines jobs, statuses, form validation, statistics, heatmap data, and SVG generation.

|Infrastructure layer
|`repository/`, `paths.rs`, `error.rs`
|Handles SQLite storage, database schema, row mapping, app data paths, and app-wide errors.

|UI layer
|`ui/`, `ui_state/`
|Builds the visible interface and holds non-database UI state such as selected panel, animation step, and calendar state.

|===

== File-by-File Description

=== `src/main.rs`

This is the entry point.

It declares the application-side modules and calls:

[source,rust]
----
app::run()
----

This is clean because `main.rs` does not contain UI code, database code, or state update logic.

To change startup behaviour, window title, theme, or app initialization, do not edit `main.rs` first. Go to `app.rs`.

=== `src/app.rs`

This is the centre of the `iced` application.

Responsibilities:

* Defines `JobTracker`, the main application state struct.
* Opens the SQLite repository on startup.
* Loads jobs from the database.
* Defines the `update` function for handling `Message` values.
* Defines the `view` function by delegating to `ui::view(self)`.
* Defines the app subscription used for stats animation ticks.
* Sets the app theme to `Theme::Dark`.

Important state fields:

[source,rust]
----
pub struct JobTracker {
    pub repository: Option<SqliteJobRepository>,
    pub jobs: Vec<Job>,
    pub form: JobForm,
    pub notice: Option<String>,
    pub active_panel: ActivePanel,
    pub animation: AnimationState,
    pub calendar: CalendarState,
    pub calendar_open: bool,
}
----

What to change here:

[cols="1,2", options="header"]
|===
|Goal |Where to change

|Change app title
|Change `.title("Job Tracker")` inside `run()`.

|Change dark/light theme
|Change `app_theme()` from `Theme::Dark` to another `iced::Theme`.

|Add new global state
|Add a field to `JobTracker`.

|Handle a new user action
|Add a variant to `Message`, then handle it in `update()`.

|Add a recurring animation/timer
|Extend `subscription()`.

|===

Current design comment:

The `update()` function is still manageable. As more features are added, it may become long. You have already started solving this by moving business actions into `app_actions.rs` and export logic into `app_export.rs`.

Responsibilities:

* Delete a job.
* Clear the form.
* Select a calendar day.

Important methods:

[source,rust]

What to change here:

[cols="1,2", options="header"]
|===
|Goal |Where to change

|Add validation-related behaviour after save
|`save_job()` or `persist_job()`.

|Change the notice text after saving
|`persist_job()`.

|Change what happens after delete
|`delete_job()` or `reload_after()`.

|Reset more UI state after saving
|`clear_form()`.

|Make edit mode open a side panel
|`edit_job()` and `JobTracker` state.

|===

Potential improvement:

`reload_after()` uses `self.repository.as_ref().unwrap()` after already assuming a repository exists. It is probably safe in the current flow, but avoiding `unwrap()` would make the app more robust.

A safer approach would be:

[source,rust]
----
if let Some(repository) = &self.repository {
    // load jobs
} else {
    self.notice = Some("Database is not available.".into());
}
----

== How to Improve the Project

=== 1. Centralise all design tokens

Right now, colours are partly centralised but not fully. Text sizes, padding, spacing, border radius, and some colours are scattered.

Add a `theme.rs` or expand `colors.rs`.

Example:

[source,rust]
----
pub const PAGE_PADDING: u16 = 24;
pub const SECTION_SPACING: u16 = 18;
pub const CARD_PADDING: u16 = 20;
pub const BORDER_RADIUS: f32 = 16.0;
----

This makes UI changes much easier.

=== 2. Remove fixed status count assumptions

`Stats` assumes four statuses. This is okay now, but it makes adding a fifth status more annoying.

A more scalable structure:

[source,rust]
----
pub struct Stats {
    pub total: usize,
    counts: Vec<(Status, usize)>,
}
----

Or use a map:

[source,rust]
----
HashMap<Status, usize>
----

Since `Status::all()` already exists, a vector is probably enough.

=== 3. Add filtering/search before the app grows

A job tracker becomes much more useful with search and filtering.

Recommended filters:

* Status
* Company
* Applied date range
* Text search over company/title/comments

=== 4. Improve the heatmap

The current heatmap is useful, but it only shows days that have activity.

Better versions:

* Calendar-like monthly heatmap.
* GitHub-style contribution heatmap.
* Grouped by week.
* Hover/selected date details.

=== 5. Improve testability of app actions

The repository trait is good, but `JobTracker` currently stores `Option<SqliteJobRepository>` specifically.

For stronger testability, you could make the app generic over a repository trait or use a boxed trait object:

[source,rust]
----
Box<dyn JobRepository>
----

However, because GUI state and lifetimes can get more complex, this is not urgent for a small app.

=== 6. Improve empty/error states

Current notices are simple strings. You could make them structured:

[source,rust]
----
enum Notice {
    Success(String),
    Error(String),
    Info(String),
}
----

Then render them with different colours.

=== 7. Add confirmation before delete

Currently, delete appears to happen immediately.

A safer flow:

* Click Delete.
* Store `pending_delete_id`.
* Show confirmation panel or inline buttons.
* Confirm delete sends final delete message.
* Cancel clears pending delete.

== Summary

This is a good small Rust GUI app. The most impressive part is the separation of concerns: the domain model, repository, UI state, app actions, and UI files are not all mixed together. That makes the app much easier to explain and extend.

The main thing I would improve next is the UI/theme architecture. Right now, the app is visually coherent, but design values are scattered across `colors.rs`, `style.rs`, `buttons.rs`, and individual UI files. Centralising those values will make it much easier to experiment with a new look, add panels, and adjust spacing without hunting through the whole codebase.

The app is already in a good state for learning `iced` because it demonstrates the core pattern clearly:

[source,text]
----
Message -> update state -> render view -> persist through repository
----

That is exactly the mental model you want for building more serious Rust GUI apps with `iced`.
